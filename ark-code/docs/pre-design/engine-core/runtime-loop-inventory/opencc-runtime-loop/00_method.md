# OpenCC engine-core 运行时闭环盘点法

## 为什么要换盘点方法

之前按模块目录盘点时，虽然能抓住主干，但会不断漏掉下面这些关键问题：

- 功能是如何被挑选的
- 功能是如何被装配成 prompt / runtime surface 的
- 功能是如何进入等待态、后台态、continuation、恢复态的
- 功能结果是如何回流到主线程或下一轮 turn 的

这说明模块目录盘点法适合做结构概览，但不足以支撑 Ark Code 后续“完整实现 + 完整验证”。

## 运行时闭环盘点法的核心问题

对于每一类能力，都按同样的问题来盘：

1. **入口**：它从哪里进入系统？
2. **筛选**：它如何被过滤、裁剪、条件激活？
3. **装配**：它如何进入 tool pool / command pool / context / attachment / prompt？
4. **执行**：它在单轮内如何运行？
5. **等待**：它会不会进入等待态、审批态、后台态、阻塞态？
6. **continuation**：它如何继续、恢复、重试、重连？
7. **回流**：它如何把结果带回主线程、下一轮 turn、或持久化表面？

## 这套文档的目标

最终不是只得到“模块清单”，而是得到：

- 一张 OpenCC engine-core 的**运行时闭环图**
- 一份 Ark Code 后续实现时的**阶段化核对表**
- 一套能扛随机抽查的**功能闭环说明**
