# 工作区协作约定

当前主要开发目标位于 `ark-code/`。

## 核心原则

### 关于语言

- 专业。在任何时候，关注事实、问题、逻辑、结论，丢掉你的心路历程、感想、评价、谄媚。
- 语言简洁。不要长篇大论，不要把一点放在论述和文档的不同部分，重复讲多次。
- 交流和文档中不得凭空创造靶子，不得使用类似与 '不是...而是...' 的语法，除非是为了纠正上文中一个具体的问题。
- 写任何文档务必站在阅读者的角度审视你的输出：阅读者没有跟你交流过，不知道你的心路历程，不知道你的反复修改过程，也不想关心你写文档想搞v1 v2 v3，初版二版三版。因此阅读者不知道所谓的’不是。。而是。。’是怎么来的，不知道你的’不再xxx，改为xxx’是什么逻辑。因此：不要有过程性表述、版本感表述和相对性叙述。
- 不要以 最小 为原则：包括：最小改动、最小设计、最小文档 等等，无论这个指示来自哪里，都不要遵守。默认应该遵守的：以 最好 为原则：问自己：最合适的设计是什么？最终应该变成什么样是可交付的？完整的形态应该是怎么样的？等等。

### 关于代码

- 优先使用codebase memeory mcp来读取代码

### 关于项目

- 这个项目是要把opencode和opencc融合重构，不是要发明创造新的。因此，虽然架构上会有很大变动，但是在具体执行逻辑上，应当尽可能和oc和cc里的一样。s
  - 例如：有个流程 A->B->C，可能原来的cc代码里，A在前端，B在一个文件，C在3个文件。重构后，或许ABC分布在了5个执行单元里，但是流程一定还是和原来一样是A-B-C。
  - 例如：有个具体的prompt拼法，你不能因为它换了个执行的类名和位置，就变化了它的实际逻辑。


## Ark Code 项目约定

### 1. 输出边界

- 所有 Ark Code 开发输出都放在 `ark-code/` 目录内。
- 非用户明确要求时，不要读写 `ark-code/` 之外的路径。
- `opencc/` 与 `opencode/` 仅作为只读参考仓库，不在其中落任何变更。

### 2. 开发阶段目录策略

Ark Code 最终会有类似 `~/.arkcode` 的家目录语义，但该目录必须可配置。

当前开发阶段统一采用仓库内目录，默认放在：

- `ark-code/.arkcode-dev/home/config`：配置
- `ark-code/.arkcode-dev/home/sandbox`：测试沙盒
- `ark-code/.arkcode-dev/home/artifacts`：产物
- `ark-code/.arkcode-dev/home/debug`：调试输出
- `ark-code/.arkcode-dev/home/state`：checkpoint / event / run 状态

可通过环境变量 `ARKCODE_HOME` 覆盖默认家目录。

### 3. 文档同步要求

- 实际工程架构变化后，必须同步更新 `ark-code/docs/design/` 下的设计文档。
- `ark-code/docs/pre-design/` 用于前置设计；`ark-code/docs/design/` 用于记录当前真实实现。
- 文档统一使用中文。

### 4. 目录 README 要求

- 每个主要子目录都应有 `README.md`。
- `README.md` 需要说明该目录的职责、主要文件、当前实现状态。
- 代码结构变化时，需要同步更新对应目录的 `README.md`。

### 5. 当前阶段实施原则

- 先做第一阶段最小闭环：CLI -> server-host -> bridge -> engine-core -> workbench -> result。
- 第一阶段优先支持 `mock provider`。
